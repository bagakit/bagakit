# Anti-Drift And Active-Mining Design

## Target

Improve `bagakit-researcher` so research outputs resist drift while still
supporting parallel topic extraction, retrieval planning, summarization,
insight synthesis, and proactive lead mining.

The core stance is:

- research is evidence production, not truth promotion
- parallel research is a track protocol, not a subagent launcher
- active mining is managed curiosity, not unlimited scope expansion

## First Principles

### 1. Anchor Before Search

Every non-trivial research pass needs a topic charter before retrieval begins.

The charter should name:

- core question
- decision or downstream use
- intended output
- in-scope areas
- out-of-scope areas
- source priority
- evidence threshold
- stop conditions

This directly prevents question drift and active-mining drift.

### 2. Preserve Parentage

Every artifact should know what it belongs to:

- source card belongs to a topic, pass, and optional track
- summary belongs to a source card
- claim belongs to one or more source cards or summaries
- insight belongs to one or more claims
- lead belongs to a charter, track, or unresolved contradiction

This prevents orphaned notes from becoming accidental conclusions.

### 3. Separate Observation, Claim, Insight, And Recommendation

Researcher should not let one markdown summary do all cognitive work.

Use distinct meanings:

- observation: what a source says
- claim: a normalized statement with evidence refs
- insight: a cross-source or cross-track interpretation
- recommendation: a proposed Bagakit action

This prevents summary drift and insight drift.

### 4. Make Curiosity Explicit

Active mining should be surfaced as leads:

- why the lead appeared
- what it might prove or disprove
- expected value
- cost or scope risk
- pursue, defer, reject, or promote status
- stop rule

This keeps proactive discovery valuable without letting it silently consume the
topic.

### 5. Merge Through Evidence, Not Conversation

Parallel workers should write durable artifacts and return lightweight refs.
The parent merges source cards, summaries, claims, insights, and lead outcomes.

This avoids the "game of telephone" failure where useful source nuance is lost
through coordinator summaries.

## Proposed Surfaces

### Topic Charter

Path:

```text
.bagakit/researcher/topics/<class>/<topic>/charter.md
```

Suggested fields:

- `question`
- `decision_use`
- `output_shape`
- `in_scope`
- `out_of_scope`
- `source_priority`
- `evidence_threshold`
- `stop_conditions`
- `drift_sentinels`

The charter is hand-authored or generated by `plan-pass`, then preserved.

### Pass Note

Path:

```text
.bagakit/researcher/topics/<class>/<topic>/passes/<pass-id>.md
```

Purpose:

- records the research pass objective
- lists tracks
- lists intended source classes
- records budget and stop conditions
- records merge expectations

### Track Contract

Path:

```text
.bagakit/researcher/topics/<class>/<topic>/tracks/<track-id>.md
```

Suggested fields:

- `parent_pass`
- `parent_charter`
- `track_question`
- `required_source_types`
- `preferred_sources`
- `disallowed_sources`
- `source_id_range`
- `owned_output_files`
- `minimum_evidence`
- `lead_policy`
- `drift_check`
- `merge_notes`

The most important field is `track_question`. It should be narrower than the
charter but must not change the charter.

### Source Card V2

Keep source cards under `originals/`, but strengthen their expected content:

- source id
- title
- URL
- authority
- published date when known
- topic/pass/track refs
- why kept
- source role: primary, implementation, benchmark, representative, contrast, or
  background
- scope fit: core, context, contrast, or deferred
- limitations

### Claim Ledger

Path:

```text
.bagakit/researcher/topics/<class>/<topic>/claims.md
```

Suggested claim fields:

- claim id
- statement
- kind: observation, inference, or recommendation
- evidence refs
- counterevidence refs
- confidence: high, medium, low, or speculative
- status: open, supported, contradicted, superseded, or promoted
- Bagakit implication

The ledger can start as one markdown file. Do not add a database until there is
clear pressure from repeated use.

### Insight Cards

Path:

```text
.bagakit/researcher/topics/<class>/<topic>/insights/<insight-id>.md
```

Suggested fields:

- insight
- source claims
- counterclaims
- confidence
- why it matters
- borrow
- avoid
- Bagakit implication
- next action

Insight cards should be required for non-obvious recommendations.

### Lead Queue

Path:

```text
.bagakit/researcher/topics/<class>/<topic>/leads.md
```

Suggested lead fields:

- lead id
- originating artifact
- hypothesis
- expected value
- suggested query or source
- status: open, pursued, rejected, deferred, or promoted
- stop rule
- outcome

This is the main proactive-mining surface.

## Command Direction

### First Slice

Extend the existing `30-parallel-research-optimization-plan.md` first slice
with anti-drift support:

- `plan-pass`
- `add-track`
- `list-tracks`
- `add-lead`
- `add-insight`
- `doctor --quality`
- `doctor --drift`
- managed-section `refresh-index`

`plan-pass` should optionally create:

- `charter.md`
- `passes/<pass-id>.md`
- `tracks/*.md`
- an empty `leads.md`
- an empty `claims.md`

### Later Slice

Add generators only after the markdown workflow stabilizes:

- `add-claim`
- `resolve-lead`
- `new-synthesis`
- `render-handoff --kind selector`
- `render-handoff --kind evolver`
- `render-handoff --kind living-knowledge`

Derived JSON may be useful later, but only as generated state. Markdown remains
the authoring truth.

## Drift Doctor Checks

`doctor --drift` should be warning-only at first.

Useful warnings:

- topic has no charter
- pass has no track contracts
- track lacks explicit output ownership
- source card lacks authority, why-kept, source role, or scope fit
- source card is `context` or `deferred` but used to support a recommendation
- summary has no source card
- claim has no evidence refs
- recommendation has no counterevidence section
- insight has only one source and is not marked low or speculative
- lead has no expected value or stop rule
- pursued lead has no outcome
- latest pass is missing from `index.md`
- index has no current synthesis for the latest pass

The check should not fail normal structural `doctor` until the workflow is
proven. The first implementation should surface quality debt without blocking
small research topics.

## Capability Map

### Topic Extraction

Implement through `plan-pass`:

- parse the user's prompt into a charter
- propose three to six tracks
- identify local prior topic materials before new external search
- name disallowed scope explicitly
- create drift sentinels

### Retrieval

Implement through track contracts and source cards:

- write planned queries before execution
- classify source type before preserving the source
- prioritize primary, official, implementation, benchmark, and representative
  sources over generic summaries
- keep freshness requirements explicit

### Summarization

Implement through stronger source summaries:

- source-specific summary remains under `summaries/`
- every summary keeps `borrow`, `avoid`, and `Bagakit implication`
- summaries that contain reusable claims should update `claims.md`

### Insight

Implement through insight cards and synthesis:

- insight cards are for cross-source or cross-track interpretation
- each insight links to claims and counterclaims
- synthesis summarizes what changed, what remains uncertain, and what action is
  justified

### Active Mining

Implement through `leads.md`:

- new leads are captured instead of silently pursued
- high-value leads can create new tracks or future passes
- low-value leads are rejected with a short reason
- deferred leads preserve future reuse without expanding current scope

## Recommended Implementation Order

1. Add directory support for `passes/`, `tracks/`, and `insights/`.
2. Add `plan-pass` with optional `charter.md`, `claims.md`, and `leads.md`.
3. Add `doctor --drift` as warning-only.
4. Add managed index sections for passes, tracks, claims, leads, and insights.
5. Extend validation to assert drift warnings for bad fixtures.
6. Add one eval topic that intentionally tries to drift into adjacent scope and
   verifies that `doctor --drift` flags it.

## Quality Bar

This design is valuable only if it stays small.

Accept:

- explicit contracts
- low-friction markdown
- warning-first quality checks
- repo-local evidence
- optional handoffs

Reject:

- hidden databases
- mandatory long-form report generation
- built-in web crawling
- built-in subagent orchestration
- automatic durable promotion
- per-topic ceremony for trivial lookups

## Bottom Line

The next version of `bagakit-researcher` should make drift visible before it
becomes truth. The minimum useful implementation is not more search. It is a
charter, track contracts, source-card quality, claim and insight provenance, a
lead queue, and a warning-only drift doctor.
