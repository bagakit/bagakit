# Interview Record Template

面试记录的价值在于让后来的人在两分钟内看明白：这个人该不该推进，强在哪里，短板是不是当前岗位的阻断项。

## 一句话原则

先写结论，再贴证据。证据按能力主题归类，时间顺序只在确实影响判断的时候保留。

## 什么时候用这份模板

这份模板适合三类场景：

- 面试过程中已经记了很多碎片，但需要整理成正式面评
- 多位面试官要共享判断，需要快速对齐“推进 / 不推进”的理由
- 候选人有一些可取点，但阻断项也明显，需要把“为什么不推进”写得具体、可复核

如果只是临时速记，可以先记原话。归档前，最好都压成这份结构。

## 填写顺序

1. 先写一句总判断。
2. 再写 2-3 个最关键判断，其中至少 1 个是阻断项。
3. 然后按能力主题收证据，不按时间顺序复述。
4. 每个主题最后压一句小结。
5. 最后补“与当前岗位要求的差距”，把结论和岗位 bar 连上。

## 模板正文

### 评分口径

如果团队已经有自己的等级定义，优先用团队口径。没有的话，可以先用这一套轻量标记：

- `4`：明显强信号，超出当前岗位预期
- `3`：基本达标，能独立说清楚
- `3-`：勉强能答，深挖明显变弱
- `2`：低于当前岗位 bar

### 按岗位裁剪

如果岗位更偏研究，可以把“研究深度 / 评测意识”展开成独立大节。岗位更偏工程时，可以加大“系统 / 业务理解”的篇幅。岗位明确强调 AI Coding 时，可以把工具使用、需求对齐和 workflow 沉淀拆得更细。

## 最容易犯的错

最常见的问题是信息不少，但判断没有被压实。

1. 开头没有结论，读完一整页还不知道要不要推进
2. 按时间顺序写流水账，导致真正的判断被问题顺序淹没
3. 只写“深度一般”“不 solid”，不写具体缺口到底是研究深度、评测意识，还是 ownership
4. 记了很多回答，但没有一句明确判断这是不是岗位阻断项
5. 写了候选人的问题，却没写“和当前团队要求差在哪”

## 推荐用法

如果原始记录很碎，先不要急着润色，先做一次“压缩”：

1. 把所有碎片先归到 3-5 个能力主题
2. 每个主题只保留最能支撑判断的问题和追问
3. 先写每个主题的小结，再回填具体记录
4. 最后把所有小结压成开头 3 行结论

这样整理出来的面评，通常更短，判断也会更明确，更方便后续复盘。
