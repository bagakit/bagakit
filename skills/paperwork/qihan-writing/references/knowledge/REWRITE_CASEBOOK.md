# Rewrite Casebook

这份 casebook 收录真实长文写作里沉淀下来的高信号改写样本。

它有两个用途：

1. 让写作者快速看到“这种句子为什么不够好，以及更好的写法是什么”
2. 让当前任务可以快速找到可复用、可比较的经典样本

这些例子来自具体项目，但这里保留它们，不是为了复制项目语境，而是为了保留最有辨识度的写作差异。

为避免把原项目语境带进 skill，这里的样例已经做过脱敏处理：保留句法差异，不保留可识别的人名和内部机制名。

## 使用方式

- 先按类别找最接近当前问题的 case
- 看原句与改写句的差异
- 看四维分析：文字 / 修辞 / 叙事方法 / 情绪
- 再看“后续怎么参考”，把它提升成可复用规则
- 最后回扫当前文稿里的同类句式，而不是只改命中的一句

## 元数据字段

每个 case 建议至少带三层元数据：

- `Tags`：这个 case 主要在解决什么问题
- `Failure Mode`：原句真正失败在哪里
- `适用边界`：这类改法适合什么场景，不适合什么场景

## A. 从作者动作回到对象本身

### Case A1

- Tags: `作者动作` `对象定义` `后台口吻`
- Failure Mode: 作者态度抢在对象本身之前，读者先看到作者情绪，再看到命题
- 适用边界: 适合长文开头和段首定义句，不适合纯个人随笔里故意保留作者情绪的场景

- 原句
  `我越来越不愿意把这套实践机制写成“某个活动做得不错”的总结。这样写太轻了，也太容易把事情写窄`
- 改写
  `这个活动的本意不是分享，甚至不是实践。建立之初，真正想构建的是一个启动程序。`
- 改写多维度分析
  - 文字:
    从主观感受起手，改成直接落对象与本意。
  - 修辞:
    从自我态度表达，改成定义式判断。
  - 叙事方法:
    从“作者怎么看”切换为“这件事是什么”。
  - 情绪:
    去掉“烦”“不愿意”这种个人情绪，保留力度但不撒气。
- 好在哪里, 后续怎么参考:
  - 读者更快进入对象本身，不会先被作者情绪拦住。
  - 凡是出现“我不想把…写成…”这类句子，都优先改成“它本来是什么 / 不是什么”。

### Case A2

- Tags: `第一人称` `强判断` `判断缓冲`
- Failure Mode: 关键判断被“我觉得”这类缓冲词削弱
- 适用边界: 适合主张句、结论句、段首句；不适合需要明确标注不确定性的事实判断

- 原句
  `这也是为什么我觉得，这套实践机制不能只当成“AI Coding 经验交流”来看`
- 改写
  `所以，这套实践机制绝对不是“AI Coding 经验交流”。`
- 改写多维度分析
  - 文字:
    去掉“我觉得”这种判断缓冲。
  - 修辞:
    从含蓄判断改成硬落锤。
  - 叙事方法:
    从解释作者观点，改成给读者定性。
  - 情绪:
    更果断，也更像成文里的主张。
- 好在哪里, 后续怎么参考:
  - 强判断应该直接写，不要总先打软垫。
  - 遇到“我觉得 / 我认为 / 我不想”这类句子，先检查能不能直接落结论。

### Case A3

- Tags: `第一人称` `经验出场` `样本宽度`
- Failure Mode: 纯主观感受出场，没有交代经验来路
- 适用边界: 适合需要作者亲身经验背书的段落；不适合纯事实综述

- 原句
  `我前几年深度使用 AI Coding 以后，一个越来越强的感受是`
- 改写
  `几年深度使用 AI Coding，从最早自己写到在多个主流工具间辗转，有一个越来越强的感受。`
- 改写多维度分析
  - 文字:
    把纯主观起手，改成带经历范围的经验起手。
  - 修辞:
    从情绪感受改成经验浓缩。
  - 叙事方法:
    把“我有感受”变成“这份感受有来路”。
  - 情绪:
    少一点抒情，多一点分量。
- 好在哪里, 后续怎么参考:
  - 作者经验可以写，但要带上时间跨度和实践轨迹。
  - 需要自我出场时，优先交代样本宽度，而不是直接抒情。

### Case A4

- Tags: `反驳` `让步结构` `因果拆解`
- Failure Mode: 反驳建立在情绪表态上，而不是命题拆解上
- 适用边界: 适合回应常见误解或对立观点；不适合纯中性介绍段

- 原句
  `这也是我越来越反感把问题写成“模型够不够强”的原因`
- 改写
  `显然，模型变强是原因，但也不是全部。它引领时代跃迁，但并不改变每个阶段都还有人类必须去创造的价值。`
- 改写多维度分析
  - 文字:
    从情绪化反感，改成因果拆解。
  - 修辞:
    从表态式反驳，改成结构化让步与收束。
  - 叙事方法:
    不再围绕作者情绪，而是围绕命题本身展开。
  - 情绪:
    少了火气，多了层次。
- 好在哪里, 后续怎么参考:
  - 想反驳一个观点时，优先使用“部分成立，但不足以解释全部”的结构。

### Case A5

- Tags: `转场` `落锤` `删除作者动作`
- Failure Mode: 用作者动作句代替真正的判断起手
- 适用边界: 适合进入核心判断前的最后一句；不适合确实需要交代作者决策过程的元写作场景

- 原句
  `我现在更愿意把这件事说得直白一点`
- 改写
  `不言自明的一点是：`
- 改写多维度分析
  - 文字:
    去掉作者的姿态动作。
  - 修辞:
    从转场口头禅改成落锤提示。
  - 叙事方法:
    让句子直接为后面的判断服务。
  - 情绪:
    更冷，更稳。
- 好在哪里, 后续怎么参考:
  - “我要开始说了”式句子，通常都可以删。
  - 需要转入核心判断时，直接落判断本身，最多用一句极短提示。

## B. 从含混定性到分层定义

### Case B1

- Tags: `对象定义` `层级拆分` `目标抬升`
- Failure Mode: 形式、关注中心、最终目标混写在一起，导致定性发虚
- 适用边界: 适合系统、活动、机制、项目的开篇定义；不适合一句话新闻摘要

- 原句
  `这套实践机制不是一次分享，也不只是几场实践。它从建立之初，瞄准的就是另一件事`
- 改写
  `定性的看，这套实践机制是“实践驱动的交流活动”。虽然以分享的形式展开，但是它的关注中心从来不是分享，而是实践自身。而不论分享还是实践，都不是它的最终目标。`
- 改写多维度分析
  - 文字:
    从并列否定，改成先给更准确的中层定义，再把目标继续上抬。
  - 修辞:
    从回避式定义，改成先定性、再分层的正面定义。
  - 叙事方法:
    先交代它以什么形式出现，再交代它真正围绕什么运转，最后把目标继续抬高。
  - 情绪:
    少一点硬拗的落锤，多一点层次清楚的稳劲。
- 好在哪里, 后续怎么参考:
  - 最好的定性，不一定是一步到顶，也可以先给一个准确中层，再继续上抬到真正目标。
  - 凡是“形式 / 关注中心 / 最终目标”混在一起的句子，都要拆层写。

### Case B2

- Tags: `概念层级` `上位概念` `知识体系`
- Failure Mode: 把常见形式误写成这一层的定义
- 适用边界: 适合方法论、体系、知识结构类文本；不适合只给实例列表的说明页

- 原句
  ``requirements / design / tasks` 只是其中最显眼的形式`
- 改写
  `这里更准确的说法，不是 requirements / design / tasks，而是以 Spec 为代表的 Knowledge Base。`
- 改写多维度分析
  - 文字:
    从举出一个过窄的形式集合，改成先给出真正的上位概念。
  - 修辞:
    从形式枚举，改成概念校正。
  - 叙事方法:
    先把层级摆对，再谈具体表现，避免用局部形态误代整体。
  - 情绪:
    去掉模糊说法，换成更硬的概念澄清。
- 好在哪里, 后续怎么参考:
  - 写体系时，先给上位概念，再给代表形态，最后给例子。

## C. 从活动安排到机制约束

### Case C1

- Tags: `机制特征` `去日历化` `组织约束`
- Failure Mode: 用时间线替代机制描述
- 适用边界: 适合活动背景、运行机制说明；不适合确实需要时间轴的纪实材料

- 原句
  `从 2025-10-24 到 2026-02-28，这期实践计划每周五固定拿出一段时间，只讲一件事`
- 改写
  `时间是每周五，而且每个人都会成为分享者。`
- 改写多维度分析
  - 文字:
    从日历信息，改成机制特征。
  - 修辞:
    从发生说明，改成组织约束。
  - 叙事方法:
    不是在记时间线，而是在写制度怎样塑造行为。
  - 情绪:
    更平静，但更有抓手。
- 好在哪里, 后续怎么参考:
  - 年月日通常不是重点，机制特征才是重点。

### Case C2

- Tags: `规则表达` `制度感` `人如何被塑造`
- Failure Mode: 规则只是静态列出，没有写出它如何作用到人
- 适用边界: 适合写组织规则、活动规则、流程约束；不适合纯 checklist 文档

- 原句
  `时间是每周五。每个人都可能成为分享者。场景必须真实，过程必须具体，结果最好能量化，失败样本不能绕开。`
- 改写
  `每周五, 每个人都会成为至少一次分享者. 场景必须真实，过程必须具体，结果最好能量化，而且鼓励失败.`
- 改写多维度分析
  - 文字:
    从抽象制度说明，改成带机制含义的活动规则。
  - 修辞:
    从静态规则，改成有执行感的表达。
  - 叙事方法:
    不只写规则，还写规则如何作用到人。
  - 情绪:
    少一点平铺，多一点制度感。
- 好在哪里, 后续怎么参考:
  - 写规则时，不只写规则本身，要写规则如何作用到人。

### Case C3

- Tags: `实践内容` `舞台动作降权` `机制表达`
- Failure Mode: 舞台动作抢走了实践内容的注意力
- 适用边界: 适合活动型文章和组织机制文章；不适合需要强调人物戏剧性的叙事

- 原句
  `每周五，都有人站出来分享；跑完整期，每个人至少讲一次`
- 改写
  `每周五都有一个人分享他最近的实践, 一轮下下每个人至少讲一次。`
- 改写多维度分析
  - 文字:
    从“站出来分享”这种偏活动现场的说法，改成“分享最近的实践”，把对象从人转回实践内容本身。
  - 修辞:
    从概括性制度描述，改成更有画面的机制表达。
  - 叙事方法:
    不再强调舞台动作，而是强调团队持续把最近的实践拿上台面。
  - 情绪:
    少一点仪式感，多一点日常运转感。
- 好在哪里, 后续怎么参考:
  - 写这类组织机制时，优先写“人把什么拿出来”，而不是只写“人做了一个动作”。

### Case C4

- Tags: `约束与机会` `文化土壤` `动力装置`
- Failure Mode: 机制只被写成规训，没有写出它如何改变人和团队
- 适用边界: 适合组织文化、机制价值、团队心智段落；不适合法规式硬规则文本

- 原句
  `这不是日程安排，而是一种组织约束。它要求每个人都把自己在现场碰到的问题、做过的判断、踩过的坑，拿出来放到团队视野里。`
- 改写
  `这是一种约束，也是一种机会。个人都得把自己在现场碰到的问题、做过的判断、踩过的坑，拿出来放到团队视野里，所有人也要形成自己的 take away。这既是个人的动力装置，也是团队的。在这个时代，更值得主动培育的，是一种可以失败、但要诚实地追求突破的文化土壤。`
- 改写多维度分析
  - 文字:
    从单向约束，扩成约束、机会、take away、文化土壤四层。
  - 修辞:
    从制度说明，改成价值揭示。
  - 叙事方法:
    不只写规则如何约束人，也写规则如何塑造动力和文化。
  - 情绪:
    少一点规训感，多一点向上生长的张力。
- 好在哪里, 后续怎么参考:
  - 当一个机制既有约束又有机会时，要把两面都写出来。

## D. 从并列动作到连续过程

### Case D1

- Tags: `推进句` `段首句` `机制演化`
- Failure Mode: 只做否定澄清，没有真正打开后文
- 适用边界: 适合段首句、转入下一阶段的句子；不适合需要明确排除某种误解的 FAQ

- 原句
  `更重要的是，这不是讲完就散的活动`
- 改写
  `分享和讨论仅仅是开始。`
- 改写多维度分析
  - 文字:
    从否定句，改成推进句。
  - 修辞:
    从被动澄清，改成主动打开后文。
  - 叙事方法:
    从“它不是什么”切到“它后面会长出什么”。
  - 情绪:
    少了说明腔，多了延展感。
- 好在哪里, 后续怎么参考:
  - 这类句子很适合当段首，用来打开机制演化。

### Case D2

- Tags: `机制角色` `比喻定义` `组成部分退后`
- Failure Mode: 只写组成部分，不写系统角色
- 适用边界: 适合说明平台、机制、活动、系统；不适合极度技术化的精确定义文档

- 原句
  `分享、评论、补充材料、后续修正，都挂在同一个地方`
- 改写
  `它是一个“活动”的种子，是分享、评论、补充、修正这一切实践行为的基石。`
- 改写多维度分析
  - 文字:
    从平铺结构，改成赋予角色和功能。
  - 修辞:
    从静态描述，改成比喻式定义。
  - 叙事方法:
    从“有哪些组成部分”切到“它在系统里承担什么作用”。
  - 情绪:
    多了一点生长感，但仍然贴地。
- 好在哪里, 后续怎么参考:
  - 描述机制时，优先写“它是什么角色”，再写“它有哪些部分”。

### Case D3

- Tags: `过程命名` `潜知识` `连续链条`
- Failure Mode: 一串动作并列出现，但读者不知道它们共同构成什么过程
- 适用边界: 适合解释机制如何运转；不适合简单流水记录

- 原句
  `分享和讨论只是开始。正文、评论、补充材料、后续修正都挂在同一个地方`
- 改写
  `分享和讨论只是一个驱动程序, 为讨论、补充、修正提供了基础, 是一个潜知识提炼的过程`
- 改写多维度分析
  - 文字:
    从静态归档描述，改成驱动程序 / 潜知识提炼这种机制定义。
  - 修辞:
    从并列动作，改成过程性判断。
  - 叙事方法:
    不再把分享、评论、补充、修正写成目录项，而是写成一个连续发生的提炼链条。
  - 情绪:
    少一点文档管理视角，多一点方法论视角。
- 好在哪里, 后续怎么参考:
  - 当一组动作是连续过程时，不要停留在并列罗列，要直接给出它们共同构成的过程名称。

## E. 从黑话与硬切句到朴实、圆融的表达

### Case E1

- Tags: `黑话词` `知识流动` `朴实动词`
- Failure Mode: 动词看起来有气势，实际上不让人看见事情如何发生
- 适用边界: 适合长文成文和公开传播稿；不适合已经约定俗成的内部操作手册

- 原句
  `一步一步写出来、讲出来、挂出来`
- 改写
  `一步一步写出来、讲出来、传递扩散出来`
- 改写多维度分析
  - 文字:
    从黑话感很重的动词，改成更朴实、也更符合知识流动方向的表达。
  - 修辞:
    保留排比，但不再靠黑话撑气势。
  - 叙事方法:
    从“把东西放出来”改成“它如何继续在人与团队之间流动”。
  - 情绪:
    少一点圈内自嗨，多一点平实的力量。
- 好在哪里, 后续怎么参考:
  - 黑话动词经常显得有劲，实际上没有对象。优先换成更朴实、更带方向的动词。

### Case E2

- Tags: `分号` `圆融句` `机制循环`
- Failure Mode: 句间关系没有写清，只能靠分号硬切
- 适用边界: 适合机制段、对照段、递进段；不适合法律条文或极度紧凑的枚举句

- 原句
  `普通分享会到这里就结束了；这套实践机制会继续向后生长。一个问题会在几周后重新出现，一条评论里的追问会长成下一场分享的正文，一次失败会变成后面的默认规则。`
- 改写
  `普通分享会到这里通常就结束了，这套实践机制则强调继续向后生长：一个问题会在几周后重新出现，一条评论里的追问会长成下一场分享的正文，一次失败会变成后面的默认规则。如此以往，经验不但没有消失，而是开始有了版本感、传递性和可追溯性。`
- 改写多维度分析
  - 文字:

    从对比式下判断，改成目标式陈述。
  - 叙事方法:
    不再把注意力放在“哪些不重要”，而是直接说明“究竟想从这些实践里得到什么”。
  - 情绪:
    少一点辩驳感，多一点建设感。
- 好在哪里, 后续怎么参考:
  - 当一段话真正想表达的是筛选标准时，直接写标准，不要靠反复否定来逼近。
  - 遇到连续两组“不是……而是……”时，都优先检查能不能改成“承认背景 + 直接写目标”的结构。

### Case K2

- Tags: `时代判断` `收束句` `持续张力`
- Failure Mode: 收束句停留在阶段复盘，没有顺势抬到更高一层的时间感
- 适用边界: 适合阶段总结、复盘、方法论结尾；不适合一次性结案汇报

- 原句
  `如果只把第一期看成一组已经发生过的分享，这件事就写浅了。第一期真正完成的，是把问题点亮，后面的工作才刚开始。`
- 改写
  `在这个时代，每一个时间点都是起点，后面的工作才刚开始。`
- 改写多维度分析
  - 文字:
    从围绕“第一期该怎么理解”的局部判断，改成更高一层的时代判断。
  - 修辞:
    从解释型过渡，改成直接落锤。
  - 叙事方法:
    不再停留在“这一期活动怎么理解”，而是把这一期放回到持续变化的时代背景里。
  - 情绪:
    少一点复盘收口感，多一点面向未来的持续张力。
- 好在哪里, 后续怎么参考:
  - 当一句话只是为了从“阶段总结”转到“继续往前做”时，可以直接把时间感和时代感抬出来。

## L. 从概念说明到使命链条

### Case L1

- Tags: `Build` `组织使命` `动员`
- Failure Mode: 只解释概念，不继续回答“所以它要求我们做什么、所以这件事为什么存在”
- 适用边界: 适合世界观段落、方法论主轴、活动使命段；不适合纯术语定义页

- 原句
  `把原本靠默契维持的知识、规则和边界，一步一步写出来、讲出来、传递扩散出来。这也是为什么 build 这个词，应该比很多人想的更早出现。AI 真正进入工程现场以后，最先被迫改变的不是代码，而是组织处理知识的方式。这套实践机制恰好把这层变化照了出来。`
- 改写
  `把原本靠默契维持的知识、规则和边界，一步一步写出来、讲出来、传递扩散出来。面向 AI 的突飞猛进，环境正在阶梯式跃迁，而且越走越快。人在每一个“当下”，还值得做的事情，我把它叫作 Build。我觉得让人动起来，成为参与 Build 的一份子，是当下最重要的 Build。所以，AI 真正进入工程现场以后，最先被迫改变的不是代码，而是人的思维模式，组织处理知识的方式。这套实践机制是为了这个 Build 而存在的。`
- 改写多维度分析
  - 文字:
    从一段相对收束的结论句，改成“知识显性化 -> 环境跃迁 -> Build 定义 -> 动员人参与 -> 落回机制使命”的完整链条。
  - 修辞:
    从提出判断，改成把判断一层层推到“这套机制为何存在”。
  - 叙事方法:
    不再只是解释 Build 的概念，而是把 Build 和人的动员、组织变化、活动使命直接连成因果链。
  - 情绪:
    少一点概念说明，多一点作者亲自下注的力量感。
- 好在哪里, 后续怎么参考:
  - 当一个命题已经足够重要时，不要停在概念解释，要继续回答“所以它要求我们做什么”“所以这件事为什么存在”。
  - 涉及 `Build` 的段落，都要优先检查有没有把“环境变化 / 人的任务 / 组织使命”三层一起写出来。

### Case L2

- Tags: `元方法` `机制名词` `下注验证`
- Failure Mode: 抽象理念没有落到更具体的机制设计上
- 适用边界: 适合组织机制、方法论、试点机制演化段；不适合概念科普

- 原句
  `这些机制背后，站着一种元方法。专题拆解不是凭空长出来的，它是从某次关键分享里自然生出来的。这是个很重要的信号。它说明这套实践机制真正关注的，从一开始就不是“这次项目怎么做”，而是“哪些方法能被团队反复拿来做事”。如果没有这种元方法意识，短分享只会留下零散经验，不会自然长出专题拆解，更不会继续长出 Spec + Skill + 实践机制 这样的团队级表达。`
- 改写
  `这些机制背后，站着一种元方法。专题拆解不是凭空长出来的，它是从那场分享里自然生出来的。这个自然生长的过程，又来自于这套实践机制最初的机制设计，也就是迭代机制，把出现的新玩法当成副本去打。这套实践机制真正关注的，从一开始就不是“这次项目怎么做”，而是“哪些方法能被团队反复拿来做事”，迭代机制代表了它的元方法意识。如果没有迭代机制，短分享只会留下零散经验，不会自然长出专题拆解，更不会继续长出 Spec + Skill + 实践机制 这样的团队体制级表达。而专题拆解的生长，本身就是一个很重要的信号，它标志着我们的下注基本正确。`
- 改写多维度分析
  - 文字:
    从“元方法意识”这种偏抽象的概括，进一步落到“迭代机制”这个更具体的机制名词上。
  - 修辞:
    从理念说明，改成“机制设计 -> 自然生长 -> 下注验证”的完整链条。
  - 叙事方法:
    不再只是说“专题拆解”是一个信号，而是把“它为什么会自然长出来”解释成机制设计本身的结果。
  - 情绪:
    少一点抽象拔高，多一点押注被验证之后的力量感。
- 好在哪里, 后续怎么参考:
  - 当一个理念背后其实已经有更具体的机制名词时，要敢于把抽象词往下落。
  - 遇到“元方法 / 机制意识 / 组织意识”这类词，都先检查有没有更具体的机制设计可以把它钉住。

## M. 从时间秩序、风险判断到范式命名

### Case M1

- Tags: `时间坐标` `层级推进` `时间表达`
- Failure Mode: 相邻时间层级打架，让读者在时间线上绊一下
- 适用边界: 适合趋势判断、阶段回顾、演化叙事；不适合纯实时报道

- 原句
  `去年的它还处在更初级的阶段，也就是上下文架构能力。再往前看，去年的 build 可能还是 prompt、workflow、IDE 使用法。`
- 改写
  `去年的它还处在更初级的阶段，也就是上下文架构能力。再往前看，更早的 build 可能还是 prompt、workflow、IDE 使用法。`
- 改写多维度分析
  - 文字:
    从“去年 / 去年”这种时间指代打架的写法，改成“去年 / 更早”两层更清楚的时间递进。
  - 修辞:
    从轻微自我重复，改成更清楚的层级推进。
  - 叙事方法:
    把“当前阶段 -> 去年阶段 -> 更早阶段”的时间坐标摆正。
  - 情绪:
    不是风格变化，而是认知秩序更干净。
- 好在哪里, 后续怎么参考:
  - 一旦句子里出现两个相邻时间层，必须检查它们是不是在同一层级上打架。

### Case M2

- Tags: `风险判断` `心理机制` `组织后果`
- Failure Mode: 只写“最危险的是什么”，没有继续写它是怎么发生的、会把组织带到哪里
- 适用边界: 适合风险段、反思段、趋势警报段；不适合轻量总结

- 原句
  `而我现在最在意的，也不只是“这套方法能不能继续成功”，而是另一个更危险的问题：组织会不会过早地误以为自己已经会了。失败并不可怕，失败至少会暴露缺口。真正危险的是把几次工具红利误认成长期能力，把阶段性手感误认成组织已经进化。一旦这样，新的 build 就不会继续发生，下一次分层来时，组织反而会掉得更快。`
- 改写
  `同时也要警醒，关注“这套方法能不能继续成功”的同时，也要持续地关注组织会不会过早地误以为自己已经会了。失败本身并不可怕，甚至会让我们更快发现问题并调整方向。但真正危险的是把工具红利误认成长期能力，把阶段性手感误认成组织已经进化，长期追高，陷在 FOMO 的心态里。一旦这样，新的 Build 就不会继续发生，下一次分层到来时，反而面临危机。`
- 改写多维度分析
  - 文字:
    从个人收束，改成面向群体的提醒，同时补进 `FOMO` 这一层心理机制。
  - 修辞:
    从单点警告，改成“成功关注 + 风险警醒”并置的双重提醒。
  - 叙事方法:
    不再只强调“误判很危险”，而是继续解释误判怎样形成，为什么会把组织推向危机。
  - 情绪:
    少一点个人忧虑，多一点面向组织的冷静警报。
- 好在哪里, 后续怎么参考:
  - 当一个风险判断已经成立时，继续往下写“它是怎么发生的”，文章的力量会明显增强。

### Case M3

- Tags: `关系定义` `共生` `主语展开`

  - 修辞:
    从定义术语，改成直接澄清关系。
  - 叙事方法:
    不再把重点放在“人怎样使用 AI”，而是把重点转回“人和 AI 处在什么关系里”。
  - 情绪:
    少一点概念说明，多一点干净的定义感。
- 好在哪里, 后续怎么参考:
  - 当一句话真正想强调的是关系本身，而不是使用动作时，要把主语展开，把动作收掉。

  - 情绪:

## O. Case Record 的方法本身

这些 case 不只是句子样本，记录方式本身也值得保留。

### 为什么这个记录格式有效

1. 它保留了原句和目标句的原始差异，而不是只给抽象原则。
2. 它把“为什么好”拆成四层，不会把写作问题全都混成“文风不好”。
3. 它天然支持横向类比：一条新样本可以帮助你快速判断当前问题属于哪一类。
4. 它非常适合做全文回扫，因为每条都已经指出“同类句子以后怎么改”。

### 推荐的进一步使用方式

1. 给每个 case 增加标签
   - 例如：`对象定义 / 第一人称 / 黑话词 / 分号 / 机制写法 / Build / 演化感`
2. 给每个 case 增加适用边界
   - 例如：适合长文、适合段首句、不适合公告文
3. 给每个 case 增加失败反例
   - 说明“什么是看似改了，其实没改到位”
4. 给每个 case 增加全文回扫提示
   - 明确这种问题通常还会在文章哪一段重复出现

## P. 从泛化断言到具体场景切入

### Case P1

- Tags: `开头` `具体场景` `泛化断言`
- Failure Mode: 用”大多数人会…”这类泛化断言开头，读者是旁观者，没有代入感
- 适用边界: 适合技术方案、方法论、架构设计类文章的开头；不适合纯学术综述

- 原句
  `大多数人在设计 AI 工程协作体系的时候，会自然地走向同一个方向：先把文档写好，再让 Agent 读文档，再去改代码。`
- 改写
  `设想现在有一个 Agent，要在 VC 的系统里完成一个跨服务的需求——比如会议室预约流程里，某个状态变更需要同时通知 VCA 后端、触发 VCC 的推送、更新妙记的日程。这个需求不复杂，但它牵涉三个团队、至少四个仓库、若干个 PSM 边界。`
- 改写多维度分析
  - 文字:
    从泛化群体断言，改成一个具体的、有名字的工程场景。
  - 修辞:
    从作者在台上讲道，改成把读者直接放进场景里。
  - 叙事方法:
    从”大多数人怎样”切到”你现在面对的是什么”，读者从旁观者变成当事人。
  - 情绪:
    少一点居高临下的概括，多一点现场感和代入感。
- 好在哪里, 后续怎么参考:
  - 开头用具体场景，后面所有判断都有了落脚点，读者不需要自己去想”这说的是我的问题吗”。
  - 凡是出现”大多数人/通常/一般来说”这类泛化开头，都先检查能不能换成一个具体的、有细节的场景。
  - 场景要有名字（具体的系统、具体的需求），不能只是”假设有一个项目”。

### Case P2

- Tags: `泛化背书` `直接判断` `群体权威`
- Failure Mode: 用”任何…都知道”这类群体权威来背书，代替作者自己的直接判断
- 适用边界: 适合技术论述、架构分析；不适合需要引用共识的学术写作

- 原句
  `但任何在大型系统里工作过的工程师都知道，这个答案是不完整的。`
- 改写
  `在商业化系统里，大量真实约束从来没有被文档化，它们活在代码里、测试用例里、某次线上事故之后加的那个 guard 里。`
- 改写多维度分析
  - 文字:
    从用群体权威背书，改成直接给出具体的证据。
  - 修辞:
    从”大家都知道”的隐性论证，改成”这是具体发生的事”的显性论证。
  - 叙事方法:
    不再靠群体共识撑判断，而是把判断本身的内容说出来。
  - 情绪:
    少一点”我在提醒你一个常识”的口吻，多一点直接说事的力量。
- 好在哪里, 后续怎么参考:
  - 群体背书是在用”别人也这么想”代替”这件事本身是对的”，两者不等价。
  - 遇到”任何…都知道 / 众所周知 / 大家都清楚”，直接删掉这层，把后面的判断内容写出来。

## Q. 从目标宣言到定位陈述

### Case Q1

- Tags: `目标宣言` `定位陈述` `重要性说教`
- Failure Mode: 用”这是…领域，也是…关键”来解释为什么重要，有说教感，读者不需要被告知重要性
- 适用边界: 适合技术方案、架构文档、设计说明的定位句；不适合需要说服外部决策者的提案

- 原句
  `我们设计 Harness，面向的就是这个目标：在商业化系统中，让 Agent 有效参与军团规模的工程协作。这是仅靠模型能力提升还无法很好解决的领域，也是在成熟系统中真正发挥 AI 作用的关键。`
- 改写
  `这个仓库就是为这个场景设计的 Harness：一个面向商业化系统、支撑军团规模工程协作的 Agent 工作现场。`
- 改写多维度分析
  - 文字:
    从”我们设计…面向…目标”的动作句，改成直接说”这个东西是什么”的定位句。
  - 修辞:
    从目标宣言，改成定位陈述。去掉”这是…领域，也是…关键”这层重要性解释。
  - 叙事方法:
    不再告诉读者”为什么这件事重要”，而是直接说”这个东西是什么、面向什么场景”。让重要性从定位本身里透出来。
  - 情绪:
    少一点领导讲话的收尾感，多一点工程师说清楚自己在做什么的干脆感。
- 好在哪里, 后续怎么参考:
  - 重要性不需要被解释，它应该从定位本身里透出来。
  - 凡是出现”这是…领域，也是…关键 / 这是…无法解决的问题 / 这是…最重要的事”，都先检查能不能删掉，直接说定位是什么。

### Case Q2

- Tags: `回答没人问的问题` `噪音句` `读者问题意识`
- Failure Mode: 回答一个读者没有问过的问题，制造噪音
- 适用边界: 适合所有正式写作场景

- 原句
  `入口文件叫 index.md 还是 README.md，不是核心问题。能不能回答这三件事，才是核心问题。`
- 改写
  `（直接删掉，进入下一段内容）`
- 改写多维度分析
  - 文字:
    这句话在回答”文件叫什么名字”这个没人问过的问题。
  - 修辞:
    用否定来强调肯定，但否定的那半句本身就是噪音。
  - 叙事方法:
    读者的问题是”协议长什么样”，不是”文件叫什么名字”。这句话打断了读者的问题链。
  - 情绪:
    显得作者在预判一个不存在的误解，有点呆。
- 好在哪里, 后续怎么参考:
  - 每写一句话，先问：读者现在的问题是什么？这句话在回答这个问题吗？
  - 凡是在回答一个读者没有问过的问题，直接删。
  - 这类句子通常来自写作过程中某个内部讨论的残留，对读者没有价值。

## R. 从结构描述到命题标题

### Case R1

- Tags: `标题` `命题式` `描述式`
- Failure Mode: 标题只告诉读者”这节讲什么”，不给读者一个有立场的判断
- 适用边界: 适合技术方案、方法论、架构设计类文章的章节标题；不适合目录索引页

- 原句
  `Harness 的工作模型`
- 改写
  `在商业化系统中落地 AI 协作：Harness 的设计逻辑`
- 改写多维度分析
  - 文字:
    从描述性标题（这节讲什么），改成命题性标题（这节的立场是什么）。
  - 修辞:
    从中性归类，改成有场景、有判断的定位。
  - 叙事方法:
    命题性标题本身就是一个有立场的判断，读者看到标题就知道作者的角度，而不是等读完才知道。
  - 情绪:
    少一点目录感，多一点论点感。
- 好在哪里, 后续怎么参考:
  - 好标题是一个命题，不是一个分类。读者看完标题应该知道作者的立场，而不只是知道这节的话题。
  - 凡是标题只是名词短语（”XX 的模型 / XX 的方法 / XX 的结构”），都先检查能不能改成”在什么场景下，XX 的什么判断”。

### Case R2

- Tags: `标题` `聚焦` `核心命题偏移`
- Failure Mode: 标题聚焦在一个推论上，而不是文章真正的核心命题
- 适用边界: 适合技术长文、方法论文章的主标题

- 原句
  `代码即 SPEC：重新理解 Agent 的工作现场`
- 改写
  `在商业化系统中落地 AI 协作：Harness 的设计逻辑`
- 改写多维度分析
  - 文字:
    从聚焦在”代码即 SPEC”这个推论，改成聚焦在文章真正的核心命题（商业化系统 + 军团协作 + Harness 设计）。
  - 修辞:
    从一个有冲击力但范围偏窄的命题，改成更准确覆盖全文论点的命题。
  - 叙事方法:
    “代码即 SPEC”只是文章的一个判断，不是文章的主题。标题应该覆盖文章的主题，不是某个推论。
  - 情绪:
    少一点标题党的冲击感，多一点准确覆盖全文的稳感。
- 好在哪里, 后续怎么参考:
  - 标题要覆盖文章的核心命题，不是某个有冲击力的推论。
  - 写完文章后，回头检查标题：标题说的是文章的主题，还是文章里某个有趣的判断？

## S. 从格式噪音到术语嵌入与金句回环

### Case S1

- Tags: `术语嵌入` `括号格式` `标题词与技术名词分工`
- Failure Mode: 技术名词直接跟在标题词后面，两者没有分工，读者不知道哪个是概念哪个是实现名
- 适用边界: 适合“中文概念 + 对应技术名词”的定义句；不适合只有一个名字的情况

- 原句
  `**硬停 BoundaryTicket**: 必须人介入才能继续。`
- 改写
  `**硬停（BoundaryTicket）**：必须人介入才能继续。`
- 改写多维度分析
  - 文字:
    技术名词放进括号，标题词和技术名词分工清晰；冒号改为中文冒号，去掉冒号前的空格。
  - 修辞:
    从“两个词并排”改成“概念（实现名）”的标准嵌入格式。
  - 叙事方法:
    括号里的名词是“这个概念在系统里叫什么”，不是概念本身的一部分。分开写，读者一眼就知道层级。
  - 情绪:
    少一点随意拼接感，多一点定义的清晰感。
- 好在哪里, 后续怎么参考:
  - 凡是“中文概念 + 技术名词”同时出现，优先用括号嵌入，而不是直接并排。
  - 冒号前不加空格，用中文冒号。

### Case S2

- Tags: `噪音命名` `未引用术语` `回答没人问的问题`
- Failure Mode: 给一个在后文不会被引用的技术名词，是噪音
- 适用边界: 适合所有正式写作场景

- 原句
  `**软停 AggressiveTicket**: AI 可以自行处理后继续。`
- 改写
  `**软停**：AI 可以自行处理后继续。`
- 改写多维度分析
  - 文字:
    删掉 `AggressiveTicket`，因为后文没有引用这个名字。
  - 修辞:
    从“给名字”改成“只给概念”。
  - 叙事方法:
    一个名字只有在后文会被引用时才值得出现。如果只是“顺手命名”，它对读者没有价值，只增加认知负担。
  - 情绪:
    少一点“我给它起了个名字”的自我满足，多一点对读者认知负担的克制。
- 好在哪里, 后续怎么参考:
  - 每次引入一个技术名词，先问：后文会用到它吗？如果不会，删。
  - 这是“回答没人问的问题”的命名变体：给一个读者不需要记住的名字，是噪音。

### Case S3

- Tags: `金句前置` `收束回响` `判断与证据分段`
- Failure Mode: 金句被埋在段落末尾，或者金句和证据混在同一段，力量打折
- 适用边界: 适合有核心判断句的段落；不适合纯事实罗列段

- 原句
  `Gate 的结果必须留下记录。每次 gate 执行后，把结果写到固定路径（比如 **.tmp/devloop/gates/<run-id>.json**）。这样做有三个价值：人可以事后审计 AI 做了哪些判断，下一个 session 可以复用上一次的 gate 结果，出问题时可以还原 AI 当时看到的状态。没有记录的 gate，等于没有 gate。`
- 改写
  `**Gate 的结果必须留下记录，没有记录的 gate，等于没有 gate。**`

  `每次 gate 执行后，把结果写到固定路径（比如 **.tmp/devloop/gates/<run-id>.json**）。这样做有三个价值：人可以事后审计 AI 做了哪些判断，下一个 session 可以复用上一次的 gate 结果，出问题时可以还原 AI 当时看到的状态。`
- 改写多维度分析
  - 文字:
    把完整金句提到段首单独加粗，证据另起一段。
  - 修辞:
    从“先证据后结论”改成“先结论后证据”，金句完整出现在最显眼的位置。
  - 叙事方法:
    金句的后半句不再作为段落收束，而是和前半句合并成完整判断，一次性给出。证据段只负责支撑，不再承担收束职责。
  - 情绪:
    少一点“读完才知道重点”的悬念感，多一点“一眼就知道这段在说什么”的清晰感。
- 好在哪里, 后续怎么参考:
  - 凡是一段话有一个核心判断句，优先把它提到段首单独成行加粗。
  - 如果金句本身有两半（“X，没有 X 等于没有 X”），把两半合并成完整判断一起前置，不要拆开放在段首和段尾。
  - 证据段只负责支撑判断，不需要再重复收束。

## T. 节奏停顿与结论前置的强显影

### Case T1

- Tags: `学术背书` `现象自说话` `删除引用`
- Failure Mode: 用学术引用来证明一个现象，但现象本身已经足够有力，引用反而打断节奏
- 适用边界: 适合工程类、方法论类文章；学术论文不适用

- 原句
  `这些判断是隐性的，通常没有显式说明。Michael Polanyi 在 1966 年描述隐性知识时说："我们能知道的，永远比我们能说出来的要多。"工程师对"触发 ≠ 完成"的感知，正是这类知识——它以经验的形式存在，从来没有被写进任何文档。`
- 改写
  `这些判断是隐性的，通常没有显式说明——它们以经验的形式存在，从来没有被写进任何文档。`
- 改写多维度分析
  - 文字:
    删掉 Polanyi 引用，把后半句直接接上。
  - 修辞:
    从“现象 + 学术背书 + 重述现象”，改成“现象本身”。
  - 叙事方法:
    读者此刻的问题是“AI 为什么会出错”，不是“隐性知识的学术来源是什么”。引用在回答一个读者没有问过的问题。
  - 情绪:
    少一点学术腔，多一点直接说事的力量。
- 好在哪里, 后续怎么参考:
  - 每次想引用学术来源，先问：读者现在的问题是什么？这个引用在回答这个问题吗？
  - 如果引用只是在“证明这个现象是真实的”，而现象本身已经够有力，删掉引用。

### Case T2

- Tags: `短句停顿` `节奏设计` `单独成行`
- Failure Mode: 核心转折句埋在段落里，对比感被稀释
- 适用边界: 适合有强对比的转折句；不适合平铺叙述段

- 原句
  `AI 没有这个直觉。它发出指令、收到成功响应，就认为操作完成了。`
- 改写
  `**AI 没有这个直觉。**`

  `它发出指令、收到成功响应，就认为操作完成了。`
- 改写多维度分析
  - 文字:
    核心转折句单独成行加粗，后续解释另起一段。
  - 修辞:
    用排版制造停顿，让前面的铺垫和后面的转折之间有呼吸点。
  - 叙事方法:
    前面铺了三个“≠”的例子，读者已经建立了“工程师知道这些”的认知，这时候一句短句单独停顿，对比感最强。
  - 情绪:
    停顿本身就是力量，不需要额外修辞。
- 好在哪里, 后续怎么参考:
  - 凡是有“前面铺垫 + 核心转折”结构，转折句优先单独成行加粗。
  - 加粗不是装饰，是让这句话成为可以被单独引用的判断。

### Case T3

- Tags: `结论前置` `证据分段` `加粗单独成行`
- Failure Mode: 结论和证据混在同一段，读者要读完才知道重点在哪
- 适用边界: 适合有核心判断 + 举例支撑结构的段落

- 原句
  `"跑通一次"和"建立闭环"的差距，往往比想象中大得多。第一次跑通，通常是因为有人旁补了关键信息……把这次跑通当成闭环，是最常见的误判。`
- 改写
  `**"跑通一次"和"建立闭环"的差距，往往比想象中大得多。**`

  `比如，第一次跑通，通常是因为有人旁补了关键信息……把这次跑通当成闭环，是最常见的误判。`
- 改写多维度分析
  - 文字:
    结论句单独加粗成一行，证据段用“比如”另起。
  - 修辞:
    先给判断，再给证据，结构清楚。
  - 叙事方法:
    原文把结论和证据混在一起，读者要读完才知道重点。改后读者一眼看到结论，再往下读是在验证它。
  - 情绪:
    结论的力量不被证据稀释。
- 好在哪里, 后续怎么参考:
  - “结论 + 举例”结构，结论优先单独成行加粗，举例用“比如”另起一段。
  - 这和 Case S3 是同一类规则的不同场景：核心判断永远先出。

## 使用要求

当你引用 casebook 里的样本时，至少要做到：

1. 先看原句和改写句的真实差异，不要只抄结论。
2. 至少带出一条项目无关的“后续怎么参考”。
3. 把这条经验回扫到当前文稿里的同类问题，而不是只改命中句。

## U. 作者自评式元话

### Case U1

- Tags: `元话压缩` `对象判断` `首屏收束`
- Failure Mode: 作者先评论“问题更具体”，对象判断还没出现
- 适用边界: 适合首屏总论句、段首句、转折句

- 原句
  `这篇文章要回答的问题更具体。`
- 改写
  `这篇文章回答的是：同一条经验，应该进 SPEC、Skill 还是 Memory。`
- 改写多维度分析
  - 文字:
    删掉“更具体”这层元评论，直接写对象。
  - 修辞:
    从作者旁白改成对象判断。
  - 叙事方法:
    读者关心的不是作者有没有把问题说具体，而是问题到底是什么。
  - 情绪:
    少一点解释腔，多一点直给。
- 好在哪里, 后续怎么参考:
  - 看到“问题更具体 / 更硬 / 更前置”这类句子，优先问：对象句能不能直接写出来？
  - 如果可以，删掉元评论。

### Case U2

- Tags: `元话压缩` `判断前置` `去作者在场感`
- Failure Mode: 作者评价“这是个很硬的判断”，判断本身却要读下一句
- 适用边界: 适合判断句、反直觉句、总结句

- 原句
  `这是个很硬的判断：SPEC 是系统模型。`
- 改写
  `SPEC 是系统模型。`
- 改写多维度分析
  - 文字:
    删掉“很硬的判断”，保留判断本身。
  - 修辞:
    强度来自命题，不来自作者替命题背书。
  - 叙事方法:
    读者拿到判断的速度更快。
  - 情绪:
    少一点“作者在教你怎么读”，多一点事实落地。
- 好在哪里, 后续怎么参考:
  - 凡是“这是个很 X 的判断/例子/问题”，都先检查能不能只留下对象句。
  - 判断强不强，交给句子本身承担。

### Case U3

- Tags: `元话压缩` `误写改直写` `去被动评价`
- Failure Mode: 用“最容易被说轻”这类句子评价对象，常见误写却没有落地
- 适用边界: 适合解释误解、误写、边界混淆的段落

- 原句
  `这一层最容易被说轻。`
- 改写
  `很多团队会把这一层写成前置沟通。`
- 改写多维度分析
  - 文字:
    把“被说轻”换成具体误写。
  - 修辞:
    从抽象评价改成可验证描述。
  - 叙事方法:
    读者立刻知道常见错误是什么，而不是只知道“这里容易被写坏”。
  - 情绪:
    少一点作者评论，多一点对象本身的失败模式。
- 好在哪里, 后续怎么参考:
  - 看到“最容易被…”句式，优先改成“把它写成……会导致……”或“常见误写是……”。
  - 如果说不出具体误写，这句话通常还没想清楚。

### Case U4

- Tags: `黑箱比喻` `机制落地` `去空转动词`
- Failure Mode: 用“被系统接住”这类词假装说清机制，实际没有交代落点
- 适用边界: 适合方法论、系统设计、知识路由类文章

- 原句
  `资产是被系统接住的痕迹。`
- 改写
  `资产是被结构化沉淀、并能被后续系统重复消费的痕迹。`
- 改写多维度分析
  - 文字:
    把“接住”换成“结构化沉淀 + 重复消费”两层具体动作。
  - 修辞:
    从黑箱比喻改成可追问的机制句。
  - 叙事方法:
    读者不再需要猜“接住”到底是什么意思。
  - 情绪:
    少一点 agent 腔，多一点架构说明的清楚感。
- 好在哪里, 后续怎么参考:
  - 看到“接住 / 接得住”时，优先追问：是谁记录、进了哪个 artefact、怎样被复用？
  - 如果回答不出来，就不要用这个词。

### Case U5

- Tags: `作者元话` `抽象强化词` `去黑话`
- Failure Mode: 用“更硬”“钉住”给抽象判断加力，结果像在评论句子本身
- 适用边界: 适合总论句、框架句、解释句

- 原句
  `这篇文章讨论的问题更前，也更硬。`
- 改写
  `这篇文章讨论的问题更前，也更靠近底层。`
- 改写多维度分析
  - 文字:
    用“靠近底层”替换“更硬”，把抽象强化词改成对象层级。
  - 修辞:
    从作者在给句子加力，改成在说明问题位置。
  - 叙事方法:
    读者得到的是判断的层级，不是作者的口气。
  - 情绪:
    少一点 agent 在场感，多一点资深作者的稳定语气。
- 好在哪里, 后续怎么参考:
  - 看到“更硬”时，优先追问：你是在说层级、边界、证据强度，还是只是在加语气？
  - 如果只是加语气，就换成可解释的对象词。

### Case U6

- Tags: `作者元话` `黑话动词` `对象直写`
- Failure Mode: 用“把这个区别钉住”这类词假装推进论证，实际在说作者怎么写
- 适用边界: 适合承上启下句、定义句、说明句

- 原句
  `可以把这个区别先钉住。`
- 改写
  `先把这个区别说清。`
- 改写多维度分析
  - 文字:
    删掉“钉住”这种黑话动词，保留动作本身。
  - 修辞:
    从“给句子制造力度”改成“把动作说清”。
  - 叙事方法:
    读者需要的是定义，而不是作者挥锤子的姿势。
  - 情绪:
    少一点舞台动作，多一点清楚感。
- 好在哪里, 后续怎么参考:
  - 看到“钉住、打穿、压住”这类词，先检查是不是在拿动作姿态替代论证推进。
  - 如果是，就换成更朴素的说明词。
