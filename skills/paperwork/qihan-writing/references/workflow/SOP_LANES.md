# Writing SOP Lanes

有了场景路由以后，还需要 SOP 分道。

同样是“写文章”，不同场景下的最小正确动作顺序不一样。这里给出每个场景的最小 SOP。

## S1_from_scratch

### 目标

从零把文章搭出来。

### SOP

1. 先确认读者与交付物
2. 把所有材料先压成 3-5 个判断
3. 先定 H2，不急着写句子
4. 先写开头和最值钱的一节
5. 再补证据和例子
6. 最后做文风收束

### 失败信号

- 上来就按材料顺序复述
- H2 没定就开始写句子

## S2_synthesis

### 目标

从多份材料里压出规律、演化和方向。

### SOP

1. 先列出样本口径和边界
2. 先决定正文、附录、术语表各自承担什么
3. 不按时间顺序写，先按共性聚类
4. 先写“真正沉淀下来的是什么”
5. 每个判断挂 1-2 个代表 case
6. 附录里补 sample frame、glossary、inventory、缺失样本说明
7. 最后检查有没有写成材料分析，以及证据链接是否对目标读者可用

### 失败信号

- 正文像材料清单
- 证据比判断多，但主轴不清
- 样本口径前后不一致
- 正文和附录没有分工
- 绝对本机路径被当成证据锚点

## S3_user_voice

### 目标

把稿子写得像用户本人会发。

### SOP

1. 先读用户的真实表达
2. 列出用户明确说不要的说法
3. 列出用户天然会说的判断句
4. 正文里先让对象和判断站住
5. 第一人称只放在该承担的判断里
6. 最后检查是否还像“成熟编辑代写”

### 失败信号

- 内容对，但没有作者在场感
- 过度克制，连该有的“我”也没了

## S4_insight_loop

### 目标

先通过问答环，把用户真正的判断压出来，再回写正文。

### SOP

1. agent 先做预分析
2. 输出分组问题 + `a / b / c / 其他` 常用选项
3. 用户部分回答即可
4. 回收强句、风格边界和更深命题
5. 判断哪些问题已经被提前回答
6. 再回到主稿重写

### 失败信号

- 问题像空问卷
- 选项像三句换皮，用户看不出真实差异
- 用户改了很多，正文却没真正抬起来

## S5_rewrite

### 目标

把用户的句子级反馈吸收成全局改进。

### SOP

1. 原样记录“原句 -> 改写句”
2. 做四维分析
3. 抽出项目无关规则
4. 回扫全文同类问题
5. 在交付里说明哪些地方已按这条经验同步回扫
6. 再复检有没有只改局部

### 失败信号

- 只改命中句
- 回答里没有说明这条经验如何影响全文

## S6_final_polish

### 目标

做最后一轮真正值钱的收束。

### SOP

1. 只改最影响读感和力度的地方
2. 优先看标题、开头、结尾、段首句
3. 打断拖长句，清理后台腔
4. 检查节奏和概念出场顺序
5. 不再大拆正文结构
6. 用 lint 做最后一轮窄验证

### 失败信号

- 为了润色反而改轻了判断
- 重新进入大规模补材料

## 跨场景公共检查

无论走哪条 lane，最后都要检查：

1. 这轮主要问题有没有真的解决
2. 有没有不必要地退回到更早阶段
3. 有没有把用户的高信号反馈只当成局部修改
4. 当前结果是否更像“写对了”，而不是“写得更长了”
