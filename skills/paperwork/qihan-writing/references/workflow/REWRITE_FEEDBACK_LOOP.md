# Rewrite Feedback Loop

当用户直接给出“原句 -> 改写句”时，这不是普通润色意见，而是高信号监督。

正确做法不是只把那一句替换掉，而是把它吸收成可复用规则，再反哺当前文稿。

## 目标

把用户的句子级改写，升级成三层东西：

1. 当前文稿的直接优化
2. 同类问题的全文回扫
3. 当前任务里可复用的改写规则

## 标准格式

每条用户改写都先按这个格式记录：

- `"原句"` 改写为 `"目标句"`
  - `改写多维度分析`
    - 文字
    - 修辞
    - 叙事方法
    - 情绪
  - `好在哪里, 后续怎么参考`

不要先总结，不要先抽象。先把原句和改写句原样记下。

## 五步工作流

### 1. 原样记录

用户怎么改，就先怎么记。

不要先“帮用户优化”为另一个版本，也不要把原句压缩成概述。原样记录是后续分析的锚点。

### 2. 多维分析

每一条都至少看四层：

- 文字：换掉了什么词，补上了什么层级，删掉了什么废话
- 修辞：从否定改成正面定义，还是从硬切改成圆融递进
- 叙事方法：从活动描述改成问题导向，还是从静态清单改成机制过程
- 情绪：是去掉了火气、规训感、解释腔，还是补上了主张感、稳感、张力

### 3. 抽出项目无关规则

记录完具体例子后，立刻追问：

- 这次改写的通用规则是什么？
- 它适用于哪一类句子？
- 它是在修正文风，还是在修正概念层级？

规则必须是项目无关的，例子可以保留项目语境。

## 4. 全文回扫

只改命中句子，不算完成。

每拿到一条改写，都至少回扫一次当前文稿里的同类问题：

- 同类句式
- 同类叙事误位
- 同类概念层级错误
- 同类黑话或节奏问题

如果用户改的是“分号硬切句”，就去扫全文所有分号和断裂短句。
如果用户改的是“不要只写约束，要写机会和文化土壤”，就去扫全文所有机制段落。

## 5. 回写当前文稿

当同一类改法已经被抽成规则后，把它真实应用回当前文稿。

- 在交付里明确说明这条经验已经影响了哪些同类句子
- 如果当前文稿里仍有明显残留，默认这轮没有完成

## 代表性例子

### 例 1：从静态动作到知识流动

- `"一步一步写出来、讲出来、挂出来"` 改写为 `"一步一步写出来、讲出来、传递扩散出来"`
  - 改写多维度分析:
    - 文字:
      `挂出来` 太像黑话，动作悬空；`传递扩散出来` 更贴近知识如何在团队里流动。
    - 修辞:
      保留排比，但不再靠黑话撑气势。
    - 叙事方法:
      从“把东西摆出来”改成“它如何继续传播”。
    - 情绪:
      少一点圈内自嗨，多一点平实的力量。
  - 好在哪里, 后续怎么参考:
    - 黑话动词经常看起来有劲，实际上没有对象。改写时优先换成更朴实、更带方向的动词。

### 例 2：从结果描述到循环沉淀

- `"经验没有散掉，而是开始有了版本感、传递性和追责性。"` 改写为 `"如此以往，经验不但没有消失，而是开始有了版本感、传递性和可追溯性。"`
  - 改写多维度分析:
    - 文字:
      `如此以往` 补上循环感，`可追溯性` 比 `追责性` 更稳。
    - 修辞:
      从结果宣告，改成过程累积后的结果。
    - 叙事方法:
      让这句不再只是结论，而是前文机制自然运行后的后果。
    - 情绪:
      少一点管理口吻，多一点机制沉淀的稳感。
  - 好在哪里, 后续怎么参考:
    - 写机制的长期效果时，优先写它如何循环累积，少用偏管理腔的抽象名词。

### 例 3：从单向约束到动力与文化

- `"这是一种组织约束。它要求每个人都把自己在现场碰到的问题、做过的判断、踩过的坑，拿出来放到团队视野里。"` 改写为 `"这是一种约束，也是一种机会。个人都得把自己在现场碰到的问题、做过的判断、踩过的坑，拿出来放到团队视野里，所有人也要形成自己的 take away。这既是个人的动力装置，也是团队的。在这个时代，更值得主动培育的，是一种可以失败、但要诚实地追求突破的文化土壤。"`
  - 改写多维度分析:
    - 文字:
      从单向约束，扩成约束、机会、take away、文化土壤四层。
    - 修辞:
      从制度说明，改成价值揭示。
    - 叙事方法:
      不只写规则如何约束人，也写规则如何塑造动力和文化。
    - 情绪:
      少一点规训感，多一点向上生长的张力。
  - 好在哪里, 后续怎么参考:
    - 当一个机制既有约束又有机会时，要把两面都写出来。

### 例 4：从静态迁移到阶段性 build

- `"面向 AI 的工程能力，已经不再只是编码能力，而是上下文架构能力。"` 改写为 `"环境正在阶梯式跃迁，而且越走越快。人在每一个“当下”，还值得做的事情，我把它叫作 Build。"`
  - 改写多维度分析:
    - 文字:
      从静态能力迁移，改成环境跃迁和阶段性 build。
    - 修辞:
      从总结式判断，改成时代背景 + 任务定义。
    - 叙事方法:
      不再只说“能力上移”，而是先回答“为什么会变，人在当下该做什么”。
    - 情绪:
      少一点管理学口吻，多一点亲历其中的紧迫感。
  - 好在哪里, 后续怎么参考:
    - 如果一个判断本质上在讲阶段变化，就不要只写成静态迁移。

## 准出标准

一次高质量的改写吸收，至少要满足：

1. 原句和改写句都被原样记录
2. 有四维分析
3. 有项目无关的泛化规则
4. 已回扫全文同类问题
5. 在交付中能说清这条规则已经怎样改变了当前文稿
