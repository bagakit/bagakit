# Insight Interview Loop

当用户的材料已经足够多，但真正的判断、主轴和文风还没有完全显性化时，直接写正文通常会偏平。

这时更好的做法，不是继续补句子，而是先跑一轮“洞察与追问”。

但要先分清：如果“浅”来自证据、理论、时代背景或样本边界缺口，不要先跑这条支线，先看 `references/workflow/DEPTH_ESCALATION_LOOP.md`。

## 这套机制解决什么问题

它主要解决三类问题：

1. **用户想得比草稿深，但没完全说出来**
2. **材料很多，但作者真正想押注的命题还没压实**
3. **需要写得像用户本人，而不是像一个成熟编辑在代写**

## 核心思想

不是先让用户自己从零回答一堆问题。

而是：

1. agent 先基于材料分析一轮
2. 给出一组带常用选项的高质量问题
3. 用户不需要答完，只需要改掉最关键的部分
4. agent 再从这些回答里，提炼更深的命题、强句、风格边界和更高阶的因果链

用户的回答不必完整。只要足够真实，就足够有价值。

## 为什么它有效

### 1. 它降低了用户表达门槛

如果一开始就让用户从零回答，很多高价值判断其实不会自然写出来。

而有常用选项时，用户更容易：

- 直接选
- 直接改某个选项
- 只在 `其他` 里补最关键的一句

这种“局部改写”往往比从零回答更能暴露真实判断和真实风格。

### 2. 它能同时拿到“信息”和“口吻”

用户在改选项或补 `其他` 时，传递的不只是观点内容，还会传递：

- 哪些词不能用
- 哪些层级必须再抬高
- 哪些地方必须更朴实
- 哪些地方允许强烈的“我”

这类信息很难靠普通访谈直接问出来。

### 3. 它给 agent 提供了高质量推演材料

当用户只改了少数几条，但改得很重，agent 反而更容易看出来：

- 用户真正想写的不是表层主题，而是更深一层的东西
- 哪些问题已经被提前回答了
- 哪些未回答的问题，其实边际价值已经不高

## 标准工作流

### Phase 1：Agent 预分析

先由 agent 基于已有材料做一轮预分析。

输出应包括：

1. 分组问题
   - 总判断
   - 体系沉淀
   - 共性经验
   - 正在做什么
2. 每个问题的常用选项
3. 为什么暂时这样判断
4. 可直接带回正文的候选强句

这一步的目标不是替用户回答完，而是先把问题空间压成一个高质量可编辑件。

## 问题模板

问答环里的每个问题，默认按这个模板输出：

```text
当前选择第{{current}}/{{total}}项：{{question_title}}，代表{{what_this_question_decides}}，有以下常用的选择
a: {{option_a}}
b: {{option_b}}
c: {{option_c}}
其他: 直接输入
```

补充规则：

1. `{{question_title}}` 要短，最好一眼就知道这题在问什么。
2. `{{what_this_question_decides}}` 要写“这题决定正文里的哪一层判断”，不要写空泛解释。
3. `a / b / c` 必须是高频、互斥、可直接选的真实选项，不要写成长段默认答案。
4. `其他` 永远保留，用户可以直接输入一句自己的说法，不需要先反驳 `a / b / c`。
5. 如果 agent 已经有倾向，可以把更可能命中的选项放在 `a`，但不要替用户预设立场。
6. 单题正文尽量控制在 6 行内，让用户能快速扫完、快速答。

## 选项设计要求

好的选项，不只是“方便回答”，还要方便把结果带回主稿。

每个问题的 `a / b / c` 至少要满足：

1. 三个选项的分歧足够真实，不能只是同一句话换皮。
2. 每个选项都能直接映射到正文的一种写法、结构或立场。
3. 至少有一个选项代表 agent 当前的高概率判断，但不能把它伪装成唯一正确答案。
4. 如果用户真正想说的话不在 `a / b / c` 里，`其他` 必须足够自然，不能让用户觉得“还得先拆你的题”。

### Phase 2：用户部分回答

用户不需要答完所有问题。

真正高价值的不是“覆盖率”，而是这几类信息：

- 用户真正认同什么
- 用户最想反对什么
- 用户觉得哪些地方写浅了
- 用户自己的世界观如何比已有选项更深
- 用户自然会怎么说

如果用户只是改了少数几个选项，或者只在 `其他` 里补了一句，已经足够。

### Phase 3：Agent 回收强信号

用户回答后，agent 需要做三件事：

1. **提炼世界观**
   - 用户真正想讲的是什么
   - 主稿的真正内核是什么
2. **提炼强句**
   - 哪些句子已经足够强，可以直接带回正文
3. **提炼风格边界**
   - 哪些说法绝对不能再用
   - 哪些说法是用户本人会自然说出来的

### Phase 4：最强大脑式深推

这一步不是继续总结材料，而是基于用户已给出的回答，再往深处推一层：

- 这组判断背后的更高命题是什么
- 它还隐含了哪些因果关系
- 哪些问题其实已经被提前回答，所以不必继续问
- 这篇文章真正的重心是不是已经变了

这里的目标不是替用户发明新观点，而是把用户已经露出来但还没完全压实的东西推到前台。

如果这一步暴露出的不是“隐含判断”，而是“基础层缺口”，就停止追问，切到 `references/workflow/DEPTH_ESCALATION_LOOP.md`。

### Phase 5：回写主稿

只有在前四步完成后，才回到正式文稿。

这时正文要吸收的，不只是信息点，而是：

- 更深的主轴
- 更准的因果链
- 更像用户本人的句子
- 更明确的风格禁区

## 交付物建议

一次完整的洞察问答环，至少可以产出两层东西：

1. `洞察与追问.md`
   - 问题 + 常用选项 + 用户选择/其他 + 追问
2. `强句回收.md` 或正文内的强句区
   - 用户已说出的高价值句子

## 适用边界

适合：

- 用户明显有自己的判断，但草稿还写不出来
- 需要把“总结稿”抬成“本人会发的文章”
- 需要同时吸收用户观点和用户文风

不适合：

- 事实信息还远远不够
- 证据、理论、时代背景或样本边界明显没补齐
- 用户只是要快速润色，不想额外投入回答
- 纯公告、纯工具文、纯格式化文稿

## 和普通问答的区别

它不是一个信息收集问卷。

它更像一个写作上的“二次启动程序”：

- agent 先把问题空间压出来
- 用户用部分回答注入真实判断和真实口吻
- agent 再把这些高信号输入回收成更深的正文主轴

## 典型失败模式

1. 问题太多，但选项太弱
   - 用户只会嫌烦，不会愿意选，也不会愿意补 `其他`
2. 选项写成三句换皮，真实分歧不够
3. 用户只答了一部分，agent却把未回答部分也当成用户立场
4. agent 只回收信息点，没有回收文风和强句
5. agent 继续写成材料总结，没有真正吸收用户已经露出来的世界观

## 使用检查

- agent 先做预分析，而不是把空问卷丢给用户
- 用户可以部分回答，不要求完整作答
- 每道题都按统一模板输出，带 `a / b / c / 其他`
- 回收阶段不仅提炼信息，也提炼风格和强句
- 最终能真实提升正文的深度、意图对齐度和风格贴合度
