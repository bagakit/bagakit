# Longform Review Rubric

适用对象：

- 技术方法论长文
- 工程复盘型长文
- 适合发博客、公众号、内部专题分享的深度稿

不适用对象：

- API 文档
- 新闻快讯
- 纯公告
- 纯个人抒情散文

## 使用顺序

1. 先过 Hard Gate
2. 再看 weighted dimensions
3. 再决定是否给 Craft Bonus
4. 最后扣 Anti-Pattern Penalty
5. 如果是对外长文，再补一轮 distribution review
6. 每次文章评审，再补一轮 audience panel review

## Hard Gate

以下任一项不满足，文章仍应视作未成稿：

- 开头 15% 内没有明确问题和核心命题
- 标题不是判断，只是主题标签
- 至少一个核心 H2 缺少例子、证据、机制解释或反例
- 存在整段内部黑话或缩写，且首次出现未定义
- 全文没有真实场景、失败案例或具体应用场景
- 结尾没有回到主命题，只是换一种说法重复前文
- 首屏只铺垫，没有在 3 段内兑现标题承诺
- 关键判断存在无依据的人群泛化或拉踩表达，且没有来源、样本边界或明确语境支撑
- 脱离聊天、repo 或作者补充解释后，无上下文读者无法在首屏理解文章主张和 stakes
- 正文把“这篇文章如何组织 / 如何展开 / 为什么这样写”当成内容主线，而不是直接写对象判断

## Weighted Dimensions

### A. 技术文章硬质量

1. 标题与核心命题：标题就是观点，首屏迅速落题，正文与标题承诺一致。
2. 读者校准与阅读收益：知道在对谁写，也知道读者为什么值得继续读。
3. 框架原创性与解释力：框架能统一现象、压缩复杂度，并能指出边界与失败模式。
4. 证据、例子与可追溯性：关键判断有案例、来源、机制或反例支撑，结论强度和证据强度匹配。
5. 可操作性与判断标准：读者读完后能据此判断、设计、排查、取舍。

### B. 长文表达质量

6. 结构推进与节奏：开头立题，中段展开，后段升级，结尾收束；每个 H2 都要有新增信息。
7. 句段清晰度与具体性：多数句子短而准，抽象判断后面跟具体对象、动作、例子。
8. 文气、声音与可引用性：文风稳定，有命名能力，有能单独摘抄传播的句子。
9. 证据校准与语气边界：群体判断、失败归因、经验推广的强度和证据强度匹配，不靠想象中的“很多人/很多团队”撑主张。

### C. 对外传播层

10. 标题、首屏与正文一致性：标题有承诺，首屏迅速兑现，正文不是另一篇文章。
11. 分享/收藏动机：读者知道为什么要转发，也知道为什么值得收藏复用。
12. 完读拉力与段落抓手：每节都会自然带出下一个问题，中途很少掉速。
13. 无上下文可理解性与跨层穿透力：小白能跟上，相关领域读者不觉得散，专家不觉得浅。

## Craft Bonus

这层不是必需项，也不能补救命题或证据硬伤。

常见 bonus 来源：

- callback / echo / circularity
- braided / montage / cross-cut structure
- analogy / metaphor / wit / wordplay
- 节奏与镜头感控制得当

判断标准：

- 巧思必须服务论证和阅读收益
- 复杂度不能大于收益
- 不允许靠作者站出来解释“我这里很巧”

## Anti-Pattern Penalty

真正该扣的不是“太清晰”，而是这些问题：

- 公式化节奏与模板骨架
- 光滑但泛化的措辞
- 总结替代综合，或回到前文但没有新增意义
- 机械完美、无 stakes、像执行记录
- citation 掉落，或来源与当前论证脱节
- 拿模糊群体当证据，或用拉踩口吻代替机制分析
- 把写作过程、段落安排或作者组织动作端上台面，挤掉正文内容本身

## Blind Review Add-on

以下问题默认不用脚本判定，要求静室主观评审：

- 无依据的人群泛化
- 拉踩表达
- 语气过界但语法正确的句子

推荐做法：

1. 把文稿或片段单独交给一个 blind reviewer。
2. 不要提前告诉 reviewer 你怀疑哪一句，也不要传你想要的修改方向。
3. reviewer 只按两个问题打分：
   - 这段有没有在替一个模糊群体下结论？
   - 这段的语气是在分析对象，还是在借群体做拉踩？
4. 有 subagent 时，优先用 subagent 做静室打分；没有 subagent 时，再用独立人工 reviewer 替代。

## Audience Panel Review Add-on

这层默认每次文章评审都要做。

标准不是“有没有人能看懂”，而是：

- 无上下文读者能否迅速入场
- 小白是否会被启发思考
- 相关领域读者是否会觉得结构清楚、问题抓得准
- 领域专家是否会尊重它的问题意识和深度

推荐做法：

1. 启动 3 个静室 reviewer：小白、相关领域、领域专家。
2. 每个 reviewer 都必须给一篇对照文章。
3. 每个 reviewer 都必须写出“相比对照文章，这篇更强和更弱在哪里”。
4. 最后把 3 份意见压成一个 synthesis，而不是简单拼接。

## Longform Review Questions

- 标题能不能被改成“X 的几个思考”而几乎不损失信息？
- 首屏 3 段内是否已经告诉读者，这篇文章到底在主张什么？
- 一个没有上下文的技术公众号读者，刷到首屏时会不会愿意继续读？
- 有没有把“怎么写这篇文章”误当成“这篇文章在讲什么”？
- 每个 H2 的首段能不能独立说明本节增量？
- 至少一个关键判断后面，有没有完整的“问题 -> 误判 -> 后果 -> 修复”链条？
- 有没有把真正的铰链句埋在长段后半？
- 有没有一两句值得被工程团队直接拿去做评审语言？
