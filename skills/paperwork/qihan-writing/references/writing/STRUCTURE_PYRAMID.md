# Pyramid Structure

## Why
读者的“注意力分辨率”有限：
- 一层少于 3 点：信息不足以形成结构，读者会觉得散。
- 一层多于 7 点：超出短时记忆容量，读者会失焦。

## Rules
1) **先定 H2 主支柱（3–7）**：整篇文章的 H2 标题数量应在 **3–7**，每个 H2 是文章的一根“主梁”。
2) **再定 H3 子结构（可选；若使用则 3–7）**：不是每个 H2 都必须有 H3。
   - 如果某个 H2 下引入了 H3，则该 H2 的直接 H3 子标题数量应在 **3–7**。
3) **必要时用 H4 下钻（可选；若使用则 3–7）**：同理，如果某个 H3 下引入了 H4，则该 H3 的直接 H4 子标题数量应在 **3–7**。
4) 少于 3：合并进上层（让这一层消失）。
5) 多于 7：拆成 2 个话题层，或引入中间层。
6) 每个标题块必须满足：
   - 一句话结论（可判真伪/可执行/可反驳）
   - 一段展开（证据/过程/例子/权衡至少占一个）

## Common mistake
- 不要把“论据/解释句”抬升成 H2 主梁。H2 只能是主结论/主问题/主决策。

## Recommended skeleton
- TL;DR（3 点）
- Problem / Current state（3–7 点）
- Judgment / Decision（3–7 点）
- Mechanism / Design（3–7 点）
- Evidence / Examples（3–7 点）
- Tradeoffs / Boundaries（3–7 点）
- Next steps（3–7 点，且含 Trigger + Metric）

## Thesis alignment test
在把一句话/一个观点提升为 H2/H3 之前，先做一次对齐检查：
- **它是不是这篇文章的核心支柱？** 如果不是，就不要占一个同级标题。
- **它更像“论据/解释/方法细节”吗？** 如果是，就放到上一级标题的段落里，或者作为 H3/H4。

例：
- “去 AI 味不能靠换词”通常是解释为什么要做系统化，而不是文章主支柱；它应该放在“问题定义/反模式”下面，而不是作为“为什么做 skill”的同级主论点。
