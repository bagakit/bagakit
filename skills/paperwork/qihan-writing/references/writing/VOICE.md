# qihan-writing Voice Rules

## 核心气质
- 犀利：不绕弯子，结论前置
- 客观：明确“证据 / 假设 / 推断”边界
- 严谨：给阈值、给反例、给回滚
- 实用：最小闭环实验优先
- 口吻：谦逊又强韧（不居功、不训导，但对标准不松）——详见 `references/writing/TONE_HUMBLE_TOUGH.md`
- 视角：当用户要求“从我角度写”，按第一人称扮演用户——详见 `references/writing/POV_FIRST_PERSON.md`

## 句子级硬规则
- 先结论后解释：每段第一句必须”可判真伪/可执行/可反驳”。
- 当一段里有一句可以单独代表本段判断的强句，优先把它前置；必要时单独成段，证据另起。
- 少形容词，多动词：用”做什么/怎么判/怎么验收”替代”很重要/很显著”。
- 避免套话：不写”本文将””我们可以看到””值得注意的是”这类无信息句。
- 避免作文腔触发词：如”到底是什么””我们来探讨””众所周知”等；用更直接的名词短语（例如”好的文风”）。
- 能正面定义就不要连用两层否定。必要时先给一个准确中层定义，再继续抬到真正目标。
- 不要替自己的句子做二次解说。像“这是个很硬的判断”“要回答的问题更具体”“这一层最容易被说轻”“把这个区别钉住”这类句子，都是作者在评价自己，而不是在写对象。优先删掉评价层，把对象判断直接写出来。
- 遇到“X 的问题更具体 / 更硬 / 更前置”或“把这个区别钉住”这种元话，优先压成对象句。常见改法是：把“这篇文章要回答的问题更具体”改成“这篇文章回答的是……”，把“这个判断很硬”改成判断本身，把“把这个区别钉住”改成“先把这个区别说清”。
- 不要给“人”下结论，除非你真有证据。像“很多团队会把这层写成…”“很多人会犯这个错误”这类句子，如果没有来源、样本边界或明确语境，就是作者在替现实世界编统计。优先改成对象失败模式，例如“把这层写成前置沟通，会导致 handoff 缺失”。
- 少用“被系统接住”“下游接得住”这类黑箱吞吐比喻。它们看起来有画面，实际上没有说清机制。优先写清动作：进入哪个 artefact、由谁维护、怎样被复用。
- 当一个概念解释了全文主轴，应该尽早出场，不要等到中后段才补定义。
- 泛化背书换成直接判断：遇到”任何…都知道 / 众所周知 / 大家都清楚”，直接删掉这层，把判断内容本身写出来。群体共识不等于判断正确，两者不能互相替代。
- 引入更高层级的宿主概念：当一个判断可以被某个已有理论框架命名时，引入这个框架能让论点更有说服力。引入的概念要真正推进判断，不是装饰。例：用 Polanyi 的隐性知识解释”为什么成熟系统的约束无法被文档化”；用 Conway 定律解释”为什么按团队划分的目录对 Agent 有额外认知负担”。
- **括号内不允许粗体**：括号的作用是补充说明，粗体的作用是强调主干。两者叠加会让读者不知道该看哪里。括号内如果需要标记技术名词，用 inline code（反引号）；如果只是普通说明，保持正文字体。
- **术语要有分工**：中文概念和技术名词一起出现时，优先写成“概念（`Term`）”或“概念，所谓 `Term`”；不要把两个名字硬并排。
- **只给会被复用的技术名词落名**：如果这个名字后文不会再被引用，它通常是噪音。

## 段落级硬规则
- 每段只回答一个问题。
- 每段必须有一个可抓住的“钩子”：指标/阈值/例子/反例/对比。
- **段落占比**：除非是对比/清单/步骤，否则优先写连贯段落；当某一节超过 200 字，至少要有 2 个自然段承载推理，而不是连续罗列。
- **列表也要有结构**：如果单层列表真有必要，默认控制在 3–7 项；少于 3 回到段落，超过 7 拆组、转表格或上升为子标题。inventory / rubric / casebook 可例外，但要分组。
- 段落要圆，不要靠分号把几个本该有因果关系的意思硬焊在一起。
- 当用户改写了一句，不只改这一句；要回扫全文里同类句式和同类叙事误位。

## 输出结构偏好
- 先 TL;DR，再展开。
- 对外长文要在首屏 3 段内兑现标题承诺。
- 研究/方案类正文默认按“分析→思考→拆解→过程→实例→落地”推进；不要上来就罗列。
- 任何建议必须落到：Next step + Trigger + Metric。
