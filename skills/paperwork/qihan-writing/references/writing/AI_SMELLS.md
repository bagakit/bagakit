# De-AI Smells Checklist

## 高频 AI 味词
- 深入/探索/赋能/打造/构建/全面/显著/极大/范式/生态/闭环（滥用时）
- crucial/vital/pivotal/robust/seamless/cutting-edge
- 挂起 / 钉死 / 挂出来（这类技术黑话动词在成文里通常显得虚）

## 句式气味
- “通过…从而…进而…”的链式空句
- “确保/提升/促进/赋能”但没有对象、机制、指标
- 过度分点、过度加粗、过度 emoji
- “本文将 / 这篇总结以 / 我在整理时 / 这里的推断 / 我们可以看到 / 值得注意的是”这类后台口吻
- 靠分号把几个缺乏因果关系的短句硬拼在一起
- 技术名词和中文概念硬并排，没有“概念 / 实现名”分工
- 给一个后文不会复用的术语命名
- 结论埋在长段后半，读完才知道重点
- 首屏只铺垫，不兑现标题承诺
- **”不是…而是…”口癖**：一篇文章里出现超过 3 次，就要全文回扫统一改掉。直接说肯定的那半句更干脆，否定那半句通常是噪音
- **作者自评式元话**：`很硬的判断`、`要回答的问题更具体`、`这一层最容易被说轻`、`把这个区别钉住`、`这种说法太...`——作者站出来评论自己的句子，读者却还没拿到对象判断。直接删掉元评论，把对象本身写出来
- **“最容易被…”句式**：`最容易被低估`、`最容易被说轻`、`最容易被误写成` 往往把对象判断写成作者旁白。更自然的写法是直接描述常见误写或常见误解
- **无依据的人群泛化**：`很多团队会...`、`很多人会...`、`大多数工程师都会...`——如果没有来源、样本边界或明确语境，就是拿想象中的群体给自己的判断背书
- **拉踩表达**：借“低估、说轻、犯错、误写”给模糊群体贴标签，但没有证据、没有对象失败模式、也没有边界声明
- **黑箱吞吐比喻**：`被系统接住`、`下游接得住`、`被流程接住`——句子像在描述机制，实际没有交代到底是谁记录、谁维护、怎么复用
- **泛化断言开头**：”大多数人会…””通常来说…””一般情况下…”——这类开头让读者是旁观者。换成一个有名字、有细节的具体场景
- **泛化背书**：”任何…都知道 / 众所周知 / 大家都清楚”——用群体权威代替自己的判断。直接删掉这层，把判断内容本身写出来
- **重要性说教句**：”这是…领域，也是…关键 / 这是仅靠…无法解决的问题”——在给读者解释为什么重要，有说教感。直接说定位是什么，让重要性从定位本身里透出来
- **回答没人问的问题**：每写一句话先问”读者现在的问题是什么”。回答一个读者没有问过的问题是噪音，直接删
- **乐观泛化的边界**：`一次解决，下次就不会再卡` 这类句子，在描述”可以被彻底解决的问题”时是准确的，不是 AI 味。AI 味的乐观泛化是把”只能降低频率”的问题也写成”一次解决”。区分方法：这个问题的解决是否依赖外部状态？如果不依赖（比如补一个静态知识），乐观表达是准确的；如果依赖（比如系统在演进），就需要加条件

## 替换策略
- 把抽象形容词替换成：
  - 机制：为什么会这样？
  - 阈值：什么情况下算成功？
  - 反例：什么时候会失败？
  - 成本：代价是什么？
- 把黑话动词替换成：
  - 更朴实的动作
  - 更具体的流向
  - 更可感的过程
- 把静态动作清单替换成：
  - 这个过程在系统里承担什么作用
  - 它为什么会继续发生
  - 它留下了什么可累积的结果
- 把“概念 + 技术名词并排”替换成：
  - `概念（Term）`
  - `概念，所谓 Term`
  - 如果后文不会复用 `Term`，就直接删掉它
- 把“结论 + 证据混写”替换成：
  - 结论句前置
  - 证据或例子另起一段
- 把泛化断言开头替换成：
  - 一个有名字、有细节的具体场景
  - 场景要有具体系统名、具体需求，不能只是”假设有一个项目”
- 把作者自评式元话替换成：
  - 直接对象句：`这篇文章回答的是……`
  - 直接判断句：`SPEC 是系统模型`
  - 直接误写句：`把这层写成前置沟通，会导致 handoff 缺失`
  - 直接展开句：`先把这个区别说清`
- 把无依据的人群泛化替换成：
  - 对象失败模式：`把这层写成前置沟通，会导致 handoff 缺失`
  - 文本失败模式：`这段把 enforcement 写成了知识层`
  - 有边界的经验句：`在这组三次讨论里，争议都卡在 rollout completion 的定义`
- 把黑箱吞吐比喻替换成：
  - `进入 SPEC`
  - `沉淀到 handoff artifact`
  - `形成稳定落点`
  - `能被后续系统重复消费`

## 快速自检
- 删掉本句后，信息是否损失？如果不损失，删。
- 这句能不能被质疑？如果不能，说明太空。
- 有没有”可落地动作”？没有就补 Next step。
- 这句是在写对象，还是在解释作者怎么写？如果是后者，通常该删。
- 这句在回答读者现在的问题吗？如果不是，删。
- 这句是在写对象，还是在评价作者自己的表达？如果是后者，删或压成对象句。
- 这句是不是在替一个模糊群体下结论？如果是，问自己：证据在哪里，边界在哪里？没有就改成对象失败模式。
- 这句是不是在用“接住”这类词假装讲清机制？如果是，把动作拆开重写。
- 这篇文章里”不是…而是…”出现了几次？超过 3 次就全文回扫。
- 这个技术名词后文还会再出现吗？如果不会，删。
- 这段的核心判断是不是要读到后半段才出现？如果是，前置。
- 这一层列表有几项？如果超过 7，拆组或改成表格/段落。
- 括号里有粗体吗？有就改：技术名词换 inline code，普通说明去掉粗体。
