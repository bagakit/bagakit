# Anti-Rationalization Table

Use this table when an agent or reviewer tries to accept weak writing by
explaining away a failed rule.

The point is not to punish variation. The point is to keep quality gates from
becoming optional when the draft gets hard.

| Rationalization | Response | Required Action |
| --- | --- | --- |
| The opening does not matter because the body has the real content. | The opening sets trust, stakes, and the first question. A strong body cannot repair a misleading or empty first screen. | Add a concrete object, problem, claim, or trust anchor in the opening. |
| This title is clear enough because it names the topic. | A topic label is not a title promise. The reader needs a judgment, conflict, boundary, or question. | Rewrite the title as a claim, decision, boundary, or reader problem. |
| The draft is shorter, so it is cleaner. | Compression is only improvement if evidence classes and actionability survive. | Compare against the baseline and record any intentional scope cut. |
| The list is precise, so prose is unnecessary. | Lists can hide missing causality, movement, and hierarchy. | Keep lists for inventory or checklist use; add connective prose where the reader needs meaning. |
| The claim is probably true from experience. | Experience can support a claim only when its boundary is visible. | Add source, sample boundary, concrete case, or mark the claim as a hypothesis. |
| The model knows the tool or API. | Tool and version claims drift quickly and should not rely on memory. | Verify current docs, source, command output, or mark the claim unresolved. |
| The audience will understand from context. | Durable writing must survive outside the chat or repo context when it is meant to be shared. | Add the missing object, stake, or term definition near first use. |
| The style sounds strong. | Forceful tone is not a substitute for mechanism, evidence, and boundary. | Replace posture with object/action/signal language. |
| The reviewer can infer the next action. | Execution-quality writing must say what changes after reading. | Add next step, trigger, owner, metric, or acceptance condition. |

If a rationalization is valid for one task, record it as an accepted deviation
with a boundary. Do not silently weaken the rule.
